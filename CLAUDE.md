# CLAUDE.md

Agentic RAG ŌĆö multi-agent retrieval pipeline on LangGraph + GigaChat + LanceDB.

## Architecture

**Fully edgeless LangGraph graph.** All routing via `Command(goto=...)` returned from nodes. No `add_edge`, no `add_conditional_edges`, no `Send`.

Entry: `set_entry_point("planner")`
Exit: `Command(goto=END)` in synthesis or give_up nodes.

**Pure RAG ŌĆö single functionality.** Every query goes through retrieval. There is
no orchestrator / complexity gate (no "answer directly without searching") and no
fallbacks: no general-knowledge answer, no answering from noise. Refusals are
honest AND evidence-based: a claim of absence requires an actual search ŌĆö
descriptions can justify routing priority, never absence ŌĆö so on the initial
turn the Planner probes the least-implausible collections instead of refusing;
`give_up` without a single search happens only when the knowledge base is empty.

The loop re-enters at **planner** ŌĆö on iteration the Planner re-routes to the
collection(s) most likely to hold the missing piece (mirrors Google RAG Engine's
loop that re-enters before its Search Plan agent). On the initial turn the
Planner never refuses an non-empty corpus unsearched: if no collection looks
relevant it emits probe routes (prompt rule + a code backstop capped at
`_PROBE_LIMIT`); the judge then rules on the actual retrieval. It goes straight
to `give_up` only with an empty knowledge base, or on iteration when every
plausible collection is searched to exhaustion.

## Nodes (6 total)

| # | Node | Returns |
|---|------|---------|
| 1 | planner | `Command(goto="query_rewriter" \| "give_up")` ŌĆö `"give_up"` only for an empty KB or iteration exhaustion; an implausible-looking corpus is probed on the initial turn, not refused |
| 2 | query_rewriter | `Command(goto="search_fanout")` ŌĆö always rewrites the Planner's routes (one task per route) |
| 3 | search_fanout | `Command(goto="sufficient_context")` |
| 4 | sufficient_context | `Command(goto="synthesis" \| "planner" \| "give_up")` ŌĆö `"planner"` re-routes for the missing piece |
| 5 | synthesis | `Command(goto=END)` |
| 6 | give_up | `Command(goto=END)` ŌĆö system refusal, no LLM |

## Key design decisions

- **No edges** ŌĆö every node returns `Command(goto=...)`, zero `add_edge` calls
- **No Send** ŌĆö fan-out parallelism is `asyncio.gather` inside `search_fanout`
- **TypedDict state** ŌĆö not Pydantic BaseModel; uses `Annotated[list, operator.add]` reducers so `search_results`/`trace`/`rewritten_queries` accumulate across iterations
- **All async** ŌĆö nodes are `async def`, tools are `async def`, streaming via `graph.astream()`
- **Only vector search tools** ŌĆö `vector_search` and `list_collections` via LanceDB; no web/Wikipedia APIs
- **Honest refusal** ŌĆö Give Up node builds a system-generated message (no LLM) listing what was found, what's missing, and why. The final answer is rendered as **markdown** in the web UI, so collection names (underscores ŌåÆ italics) are backtick-wrapped everywhere they reach the user: give_up wraps its lists and any bare names inside the judge-written reason (`_backtick_names`); Synthesis sees names pre-backticked in its prompt (inventory + source headers) plus an explicit rule, and mirrors the format
- **Structured output via function calling** ŌĆö `get_structured_llm()` in `common.py` uses `with_structured_output(schema, method="function_calling")` ŌĆö GigaChat's native function-calling path; kept explicit so every structured node goes through the same mechanism regardless of library defaults.
- **Validation-driven retries + honest LLM-failure refusal** ŌĆö every structured node calls `generate_structured(schema, prompt)` (`common.py`), not the raw LLM. Every requirement on a result is a Pydantic constraint (required fields; `RouteStep._non_empty` rejects blank `collection`/`subquery`; `SufficientContextResult._insufficient_verdict_is_actionable` rejects a `sufficient=False` verdict whose `feedback` is empty ŌĆö its presence is a code contract, the Planner's iteration mode keys off `bool(feedback)` ŌĆö or whose `missing_parts` have blank items; feedback's *form* is deliberately NOT validated: its only consumer is the Planner-LLM reading prose, and a label check would turn good-content-worded-differently into re-prompts that can escalate a correct verdict into give_up). Constraints needing node-time context use the same path: `make_sufficient_context_schema(collection_names, query)` (`state.py`) returns a dynamic subclass (function-calling name kept stable) with two extra validators ŌĆö `question_verbatim` must be a (normalized-)literal copy of the user's query, and `feedback`/`missing_parts` of an insufficient verdict must not contain any collection name (weak models prescribe routes regularly, so the ban is mechanical, not prompt-only). A violation ŌĆö or a transport error, or no tool call ŌĆö raises, and `generate_structured` re-prompts with the failure text up to `STRUCTURED_MAX_RETRIES` times (one uniform path: no per-schema retry hooks). If the model still can't satisfy the schema it raises `StructuredGenerationError`; the `llm_failsafe` wrapper (applied to every node except `give_up` in `build_graph`) catches that **and** GigaChat transport errors (`gigachat.exceptions.GigaChatException`, `httpx.HTTPError`) from the free-text nodes (`query_rewriter`/`synthesis`), routing to `give_up` with `llm_error` set so the refusal honestly cites the model failure instead of crashing. Non-LLM exceptions (code bugs) propagate.
- **All LLM-facing text is Russian** ŌĆö agent prompts, schema field `description`s (they ship inside the function-calling schema), validator error messages (they are re-prompted back to the model on failure), and the placeholder strings interpolated into prompts (`(ą▒ąĄąĘ ąŠą┐ąĖčüą░ąĮąĖčÅ)`, `(čĆąĄąĘčāą╗čīčéą░čéąŠą▓ ą┐ąŠąĖčüą║ą░ ą┐ąŠą║ą░ ąĮąĄčé)`, ŌĆ”). GigaChat is Russian-native and the corpus is Russian ŌĆö prompting in English made the model reason in English about Russian documents. Code identifiers, comments, logs and docs stay English.
- **Schema-Guided Reasoning (field order = reasoning order)** ŌĆö structured output is generated field-by-field in declaration order, so a schema's fields must read like a person's reasoning chain with the verdict near the end, not the start. `SufficientContextResult` (`state.py`) is ordered `question_verbatim ŌåÆ reason ŌåÆ draft_answer ŌåÆ missing_parts ŌåÆ sufficient ŌåÆ feedback`: first a verbatim **copy** of the user's question (copying is reliable even for weak models, unlike generating sub-questions ŌĆö and code verifies the copy against `state["query"]`), re-anchoring the judge on the question **as asked** after tens of kB of chunks inflated its idea of what was asked; then the analysis, the draft, the gaps, and only **then** the `sufficient` boolean ŌĆö grounded in the reasoning it just generated. When `sufficient` was the *first* field, the judge committed to the boolean up front and rationalized it ŌĆö letting "not found" drafts pass as `sufficient=True`. `feedback` comes last: the information gap is described only after insufficiency is concluded. The field `description`s also carry the semantics (the verdict's question is "would one more search improve the answer to `question_verbatim` as asked"; an exhausted corpus with thin findings is sufficient, a zero-findings draft never is).
- **Sufficiency = retrieval exhaustion, not answer quality** ŌĆö the judge's one question is *"would one more search of THIS corpus materially improve the answer to the question as asked?"*, never a grade against an ideal answer. `sufficient=True` in two cases: the answer is found, **or** the corpus is exhausted on the topic (every plausibly-relevant collection searched to diminishing returns ŌĆö new chunks that are off-topic count as exhaustion too) and the draft honestly collects what little there is: *┬½ą▓ ąĖčüčéąŠčćąĮąĖą║ą░čģ ąŠą▒ čŹčéąŠą╝ ąĄčüčéčī čéąŠą╗čīą║ąŠ ŌĆ”┬╗* is a full answer, routed to Synthesis. Zero findings are **never** sufficient ŌĆö that path stays an honest refusal (this is NOT the rejected "synthesis instead of give_up at max iterations" bypass: the judge itself rules, verdict and route stay consistent). Underspecified queries (┬½ą┐čāčłą║ąĖąĮ┬╗) mean "what does the corpus say about X" ŌĆö the judge must not inflate them into a questionnaire (ą▒ąĖąŠą│čĆą░čäąĖčÅ/ą┤ą░čéčŗ/čéą▓ąŠčĆčćąĄčüčéą▓ąŠ) the user never asked for. The judge prompt deliberately does NOT receive the iteration counter (budget is code's job; showing it invites "last iteration, so accept" gaming) nor the previous `missing_parts` (echoing them re-anchored the judge on its own earlier inflation).
- **Judge Ōåö Planner separation of concerns** ŌĆö the judge speaks the language of *information*: what was asked, what was found, which fact is missing. `feedback` (when insufficient) is guided toward the template *┬½ąØąĄ čģą▓ą░čéą░ąĄčé: ŌĆ”. ąØą░ą╣ą┤ąĄąĮąŠ ą▓ą╝ąĄčüčéąŠ čŹčéąŠą│ąŠ: ŌĆ”. ąÉą╗čīčéąĄčĆąĮą░čéąĖą▓ąĮčŗąĄ čäąŠčĆą╝čāą╗ąĖčĆąŠą▓ą║ąĖ: ŌĆ”.┬╗* (prompt/description-level recommendation ŌĆö only its presence is validated, see above) ŌĆö and `missing_parts` are short noun phrases naming absent facts. Collection names and "search in X" prescriptions are **validation errors** (see the bound schema above): an early version had the judge name the next collection, which made the Planner a mere executor ŌĆö it obediently re-searched a known-empty collection twice and couldn't apply its own "prefer unsearched" rule. The judge may still *consider* unsearched plausibly-relevant collections (a negative verdict isn't final while one exists ŌĆö that's a verdict criterion, so the judge keeps the inventory), but *which* collection to search next is decided by the Planner alone. Alternative phrasings are the judge's to give (only it read the chunks and saw which wording failed); the Planner carries them into `subquery`, the rewriter into the final search query.
- **Mechanical search statistics (code computes, the model reads)** ŌĆö a weak model can't reconstruct "what has been searched" from collection tags scattered across the retrieved chunks (it hallucinated "not searched yet" for a collection searched on iteration 1). `collection_search_stats` (`agents/common.py`) walks the accumulated `search_results` and the nodes inject pre-formatted ground truth: the **judge** gets the searched set + last-search novelty delta (*┬½ąŠą▒čŗčüą║ą░ąĮą░ 3 čĆą░ąĘą░, ą┐ąŠčüą╗ąĄą┤ąĮąĖą╣ ą┐ąŠąĖčüą║ ą┤ą░ą╗ +0 ąĮąŠą▓čŗčģ čćą░ąĮą║ąŠą▓┬╗* ŌĆö `+0` is the diminishing-returns exhaustion detector that kills the "let's search there again" loop) **plus the executed queries per collection** (grounds its "is there an untried angle?" call and keeps its ┬½ą░ą╗čīčéąĄčĆąĮą░čéąĖą▓ąĮčŗąĄ čäąŠčĆą╝čāą╗ąĖčĆąŠą▓ą║ąĖ┬╗ from repeating tried ones); the **Planner** gets per-collection coverage (*┬½ąĖąĘą▓ą╗ąĄčćąĄąĮąŠ 28/210 čćą░ąĮą║ąŠą▓ (13%)┬╗*, totals via `count_chunks` in `vectordb/tools.py`) **plus the same novelty delta** as its stop signal, framed as routing input, never a verdict criterion ŌĆö its prompt spells out that LOW coverage does NOT mean "barely explored, dig the same spot again" (a similar query re-returns the same top chunks; the ą¤čāčłą║ąĖąĮ trace burned 3 iterations on exactly that misreading), and that with every plausible collection exhausted it must return EMPTY steps (ŌåÆ honest give_up with an exhaustion-specific reason) instead of re-visiting. The judge deliberately gets no percentages ŌĆö a weak model reads "13% seen" as "87% unexplored ŌåÆ keep searching". The **rewriter** gets the already-executed queries for its collection (`_queries_already_tried`) so iteration rewrites stop converging to the same bag of words. For this record to be complete, `search_fanout` keeps **every** executed search in `search_results`, empty ones included (an empty search is the strongest exhaustion signal); renderers (judge/synthesis/give_up) skip chunkless entries.
- **Logging** ŌĆö `logged_node` decorator (in `common.py`, applied centrally in `build_graph`) is the single point that emits each node's `trace` entries as `logging` records under the `agentrag` logger. `setup_logging()` (`src/logging_setup.py`) is called by both `src/main.py` and `web/app.py`, so node decisions appear identically under CLI and web. `main.py` prints only the final answer (program output, not a log); the web UI's live trace stream is a separate channel.
- **Per-step token metering** ŌĆö `logged_node` also meters token usage: it installs a fresh sink (a `_token_sink` `ContextVar`) before each node runs, and `_TokenUsageHandler` (an `AsyncCallbackHandler` attached to every LLM via `get_llm`'s `callbacks=`) adds each call's `prompt_tokens`/`completion_tokens` into the current sink. Works for **structured** calls too (function-calling returns a parsed object with no `usage_metadata`, so a callback is the only place to catch them), and sums across **concurrent** calls (a node's `asyncio.gather` subtasks copy the context, so they share the sink set before the gather). Totals are stamped onto each trace entry as `input_tokens`/`output_tokens` ŌåÆ logged inline (`[in=ŌĆ” out=ŌĆ”]`) and shown in the web UI as a second line under each step plus a per-message ╬Ż total.

## State accumulation & DB scope

- `search_results`, `rewritten_queries`, `trace` use `operator.add` reducer ŌĆö each Command.update appends. Critical for the iteration loop.
- `search_results` records **every executed search, empty results included** ŌĆö it is the source the mechanical statistics are computed from (searched set, novelty delta, coverage). Entries that errored carry `"error"` and are excluded from the stats; consumers that render context (judge/synthesis/give_up) skip chunkless entries themselves.
- `search_tasks` (no reducer, overwrite) carries `[{collection, query}]` for the current turn ŌĆö one concrete-collection task per Planner route. No `collection=None` / search-all mode (pure RAG, no broad fallback).
- `db_path` (set by `make_initial_state(db_path=...)`, default None=global `LANCE_DB_PATH`) scopes every search to one LanceDB ŌåÆ per-project isolation. Threaded into `vector_search`/`list_collections`.
- `stitch_settings` (set by `make_initial_state(stitch_settings=...)`, default None) ŌĆö per-project neighbor-stitching overrides `{expand_padding, bridge_gap}` read by `search_fanout`; None / missing keys ŌåÆ global `vdb_settings`. The web threads the project's saved values here (`web/chat.py`); the CLI passes nothing.

## Multi-file search

`planner` builds one route per relevant collection; `query_rewriter` emits one `search_task` per route; `search_fanout` searches every `(collection, query)` pair in parallel (`asyncio.gather`). This is what makes cross-file multi-hop work.

## Vector DB (LanceDB)

**LanceDB** ŌĆö embedded/serverless (no DB process), stores Lance columnar files on disk, async, persists across restarts. Self-contained module at `src/vectordb/`:
- `embeddings.py` ŌĆö FastEmbed (ONNX, `paraphrase-multilingual-MiniLM-L12-v2`, **384d**, multilingual incl. Russian ŌĆö an English-only model blinds retrieval on a non-English corpus); `embed`/`embed_batch` run sync ONNX off the loop via `asyncio.to_thread`; model cached `@lru_cache`. The ~252MB model file lives in `EMBEDDING_CACHE_DIR` (default `./data/fastembed_cache`) ŌĆö FastEmbed's own default is `{tempdir}/fastembed_cache`, and /tmp is tmpfs on many distros: wiped each reboot ŌåÆ re-download every boot, broken air-gapped runs.
- `client.py` ŌĆö `get_async_db(db_path)` / `get_sync_db(db_path)`; `db_path or LANCE_DB_PATH`.
- `config.py` ŌĆö `VectorDBSettings` (pydantic-settings, `vdb_settings` instance): all vectordb knobs from `.env` (path, model, chunking, search, stitching). See [Configuration](#configuration).
- `describe.py` ŌĆö `describe_document(text)`: LLM reads an excerpt at index time ŌåÆ a 1ŌĆō2 sentence content summary. Self-contained (builds its own GigaChat client from `general_settings`, no `agents` import).
- `descriptions.py` ŌĆö JSON sidecar (`{db_path}/_descriptions.json`, `table ŌåÆ {file, description}`) storage; `load_descriptions`/`save_descriptions`. Written at index time, read by the Planner.
- `tools.py` ŌĆö `vector_search(query, collection, top_k, db_path)` and `list_collections(db_path)` as LangChain `@tool`s. **Async LanceDB gotcha**: `search()` is a coroutine ŌĆö `q = await table.search(vec)` then `await q.limit(k).to_list()`. Returns chunk `text` + `_distance` (L2, default metric) + `seq` (chunk position). `gather_neighbors(collection, hit_seqs, ŌĆ”)` does the neighbor stitching (filter-scan by `seq`, no vector). `count_chunks(collection, db_path)` ŌåÆ total rows (or None), feeding the Planner's coverage statistic.
- `indexer.py` ŌĆö `index_documents(dir, db_path, settings=ŌĆ”)` (from-scratch run over a dir) delegates to `index_files(files, db_path, fresh=ŌĆ”, settings=ŌĆ”)`; `fresh=False` is the **incremental** mode (web file edits): existing tables stay, each given file's previous table ŌĆö located via the sidecar's fileŌåÆtable mapping, `safe_table_name(stem)` fallback for legacy sidecar-less tables ŌĆö is dropped up front and rebuilt, and the sidecar is merged, not overwritten; `remove_files_from_index(names, db_path)` drops deleted files' tables + sidecar entries. `settings` = per-corpus overrides resolved by `resolve_index_settings` ŌĆö absent/None keys fall back to `vdb_settings` (.env stays the default). The resolver covers six knobs: the four index-time ones (`INDEX_TIME_KEYS` ŌĆö `chunk_size`, `chunk_overlap`, `descriptions_enabled`, `describe_max_chars`; changing any invalidates stored chunks ŌåÆ full reindex) plus the two search-time stitching ones (`expand_padding`, `bridge_gap`) that ride along in the same per-project dict but are consumed by `search_fanout`, not the indexer. Hybrid extraction (LiteParse for PDF/DOCX/PPTX, `read_text` for TXT/MD) ŌåÆ `clean_text` (collapse ragged whitespace) ŌåÆ `split_text` (RecursiveCharacterTextSplitter, chunk_size/overlap chars, splits on paraŌåÆlineŌåÆsentenceŌåÆword, never mid-word) ŌåÆ `embed_batch` ŌåÆ rows `{text, vector, seq}`. CLI: `python -m src.vectordb.indexer --dir docs/sample_docs`. LiteParse OCRs scanned/image pages with built-in Tesseract by default; set `OCR_SERVER_URL` (+ `OCR_LANGUAGE`) to delegate to a local EasyOCR/PaddleOCR sidecar via `_get_parser()` ŌĆö better Cyrillic, silences Tesseract's "Image too small to scale!!" native stderr noise.

**Schema & layout:** one **file ŌåÆ one table** (collection); rows are `{text: str, vector: float[384], seq: int}` (`seq` = chunk index in the document, enables neighbor stitching). Table name = sanitized file stem via `safe_table_name()` (LanceDB allows only `[A-Za-z0-9._-]`; Cyrillic transliterated, hash fallback, collisions disambiguated ŌĆö in incremental mode against the *existing* table set, not just the current run's). A (re)indexed file's table is always `drop_table` + `create_table` (no row-level upsert), but indexing itself is per-file incremental: only the files passed to `index_files` are touched. No ANN index built ŌåÆ exhaustive search (fine at doc scale).

**Neighbor stitching (deterministic context expansion):** vector search returns the top-k *most similar* chunks, but a contiguous structural block (table of contents, reference list) splits across chunks where only the head ranks high ŌĆö the tail falls below top-k and the answer truncates. After KNN, `search_fanout` calls `gather_neighbors`: each hit's `seq` becomes a window `[seq-EXPAND_PADDING, seq+EXPAND_PADDING]`; windows merge when the uncovered gap between them is `Ōēż BRIDGE_GAP` (effective merge distance `2*P + gap + 1`); every chunk in the merged ranges is fetched by `seq` filter-scan and the result is seq-ordered, capped at `MAX_EXPANDED`. Padding and gap are per-project overridable (state `stitch_settings` ŌåÆ `gather_neighbors` kwargs); `MAX_EXPANDED` stays global. Legacy tables without `seq` no-op (reindex to activate). `sufficient_context` shows each chunk seq-tagged so the judge sees contiguity and gaps.

**Per-file descriptions:** at index time `describe_document` (LLM) summarizes each file in 1ŌĆō2 sentences; stored in `{db_path}/_descriptions.json`. The Planner reads them via `list_collections_described(db_path)` ŌåÆ `[{collection, description}]`, so routing sees a content summary, not just the table name. Legacy tables (indexed before the feature) ŌåÆ empty description; reindex to populate.

**Corpus inventory ŌåÆ judge & synthesis:** the same `list_collections_described` list is also injected (as `get_inventory_str` in `agents/common.py`) into the **Sufficient Context** and **Synthesis** prompts as the *complete, authoritative* inventory of the knowledge base. This closes an epistemic gap: vector search returns similar chunks but never proves it has seen every document, so for "describe/list ALL files"-type queries the judge could never confirm completeness and the loop always ended in `give_up`. With the ground-truth inventory the judge can confirm full coverage (every collection searched or summarized) and Synthesis can describe every file from its summary even where chunks are thin.

The judge uses the inventory **two ways**, and the prompt must keep them distinct (an early version conflated them and made the judge mark a "not found" answer as `sufficient=True`): (1) *"all files"* questions ŌåÆ coverage is complete once every collection is searched or summarized ŌĆö don't demand proof of more documents. (2) *Specific* questions ("what is X") ŌåÆ if the answer isn't in the retrieved chunks, compare the inventory against the **code-computed search statistics** (the judge receives the searched set explicitly ŌĆö reconstructing it from the collection tags on result blocks is exactly what a weak model gets wrong) and, if any unsearched collection could plausibly hold the answer, return `sufficient=False` ŌĆö *without naming it*: the existence of an unsearched plausibly-relevant collection means the negative verdict isn't final yet, but choosing the next collection is the Planner's job (naming one in `feedback`/`missing_parts` is a validation error). A negative answer is only final once every plausibly-relevant collection has actually been searched. The **Planner's iteration prompt** receives the same statistics enriched with coverage (`ąĖąĘą▓ą╗ąĄčćąĄąĮąŠ K/N čćą░ąĮą║ąŠą▓ (P%)`) ŌĆö it never sees `search_results`, so without an explicit searched-set it would re-route to a known-empty collection and burn iterations.

**Storage & isolation:** everything under `data/` (auto-created by `get_async_db`/`get_sync_db` via `mkdir(parents=True)` and by `ProjectStore`). CLI uses global `LANCE_DB_PATH` = `./data/lancedb/_cli` (`_cli` can't collide with project UUIDs and isn't listed as a project). Web gives each project its own dir `data/lancedb/{project_id}/` and threads that `db_path` through state ŌåÆ `vector_search`/`list_collections`, so a project searches only its own files. File edits update the index **incrementally** (only the delta is re-embedded); full wipe-and-rebuild of `data/lancedb/{id}` happens only when the project's indexing settings change. Data persists between runs.

## web module (NiceGUI)

`web/` is a sibling top-level package ŌĆö absolute imports `from src... import ...`, one-directional (`web ŌåÆ src`). Run: `python -m web.app`.
- `app.py` ŌĆö NiceGUI UI passed as `ui.run(root=index)` (script mode needs a root function, not `@ui.page`). Green theme; chat freezes (blue + tremble via inlined `static/style.css`) while a project (re)indexes. `commit_edit` computes the staged-edit **delta** (`added` = new uploads/replacements/rename targets, `removed` = deletions/rename sources) and triggers the incremental update ŌĆö no changes ŌåÆ no reindex at all. The project Ōŗ« menu's first item, **Indexing settings**, opens a project-styled dialog (`.settings-card`) over the six per-project hyperparameters (four index-time + two stitching); a red Apply button is hidden until the values actually differ from the saved ones (revert hides it again) and its label tells the cost: *Apply ŌĆö full reindex* when any `INDEX_TIME_KEYS` value changed (apply validates `chunk_overlap < chunk_size`, saves via `set_index_settings`, runs `trigger_reindex`), plain *Apply* for stitching-only changes (save only ŌĆö they apply from the next search); plain close discards.
- `projects.py` ŌĆö `ProjectStore`: filesystem CRUD. `data/projects/{id}/{meta.json,files/}` + `data/lancedb/{id}/`. File list read from disk (no drift). `get_index_settings`/`set_index_settings`: per-project indexing hyperparameters under `meta.json`'s `index_settings`, always resolved through `resolve_index_settings` so unset projects get the `.env` defaults.
- `runtime.py` ŌĆö `GRAPH` (built once), `STORE`, per-project status + `asyncio.Lock`. Progress: `start_progress` (full reindex ŌĆö everything pending) vs `init_partial_progress` (incremental ŌĆö only the delta spins; untouched files show done, earlier failure flags survive).
- `indexing.py` ŌĆö `reindex_project(id)`: wipe `data/lancedb/{id}`, re-run `index_documents` (used on settings apply ŌĆö old chunks were cut with old settings); `update_project_index(id, added, removed)`: incremental ŌĆö `remove_files_from_index` for removed, `index_files(fresh=False)` for added. Both pass the project's `get_index_settings` and hold the per-project lock; status `reindexing`ŌåÆ`idle`.
- `chat.py` ŌĆö `run_chat(project_id, query)`: fresh `thread_id` per message (no state bleed), streams `(trace|answer)` from `graph.astream`. Threads the project's stitching overrides into `make_initial_state(stitch_settings=ŌĆ”)` so `search_fanout` expands hits with the project's `expand_padding`/`bridge_gap` instead of the globals.

## GigaChat API

Sber endpoint at `https://gigachat.devices.sberbank.ru/api/v1` via the official `langchain-gigachat` integration. Model: `GigaChat-2-Max` ŌĆö the freshest API-available flagship (GigaChat Ultra is chat-only, no API yet). Auth: `GIGACHAT_CREDENTIALS` (base64 authorization key from developers.sber.ru) + `GIGACHAT_SCOPE`; the SDK exchanges the key for a 30-min access token and refreshes it automatically. `GIGACHAT_VERIFY_SSL_CERTS=false` by default ŌĆö the API serves certs from the RU Ministry of Digital Development CA, absent from standard trust stores. **Temperature gotcha**: GigaChat rejects `temperature=0` (allowed range `(0, 2]`) ŌĆö `get_llm` translates a non-positive temperature into `top_p=0`, the documented deterministic mode. LLM factory cached via `@lru_cache` in `src/agents/common.py`.

**Transient-error retries (tenacity)** ŌĆö `src/llm_retry.py` is a neutral module (importable by both `agents` and `vectordb` without breaking describe.py's no-`agents` rule) exposing `ainvoke_with_retry(runnable, input)`: a tenacity `AsyncRetrying` policy used at every LLM call site (`generate_structured`, `query_rewriter`, `synthesis`, `describe_document`). Retries **only** transient errors ŌĆö `gigachat.exceptions.RateLimitError` (429: the free PERS scope allows one concurrent request, and `query_rewriter`'s `asyncio.gather` over routes exceeds that), `gigachat.exceptions.ServerError` (5xx), `httpx.TransportError` (connection drops, timeouts) ŌĆö with exponential backoff + jitter (initial `GIGACHAT_RETRY_BACKOFF_FACTOR`, cap 60s), honoring a 429's `Retry-After`; `GIGACHAT_CONNECTION_RETRIES` attempts (default 3, `0` disables). Auth/4xx/validation fail fast. Exhausted retries reraise ŌåÆ `llm_failsafe` ŌåÆ honest `give_up`. Distinct from `STRUCTURED_MAX_RETRIES`, which re-prompts on *semantic* (schema-validation) failures ŌĆö tenacity absorbs transport noise inside each semantic attempt.

## Configuration

Settings are **pydantic-settings** `BaseSettings` classes ŌĆö typed, validated, read from `.env` / process env (env var = UPPERCASE field name, case-insensitive). Two objects, two scopes:

- **`general_settings`** (`src/config.py`) ŌĆö GigaChat + agent loop: `GIGACHAT_CREDENTIALS`, `GIGACHAT_SCOPE`, `GIGACHAT_MODEL`, `GIGACHAT_BASE_URL`, `GIGACHAT_VERIFY_SSL_CERTS`, `GIGACHAT_CONNECTION_RETRIES`/`GIGACHAT_RETRY_BACKOFF_FACTOR` (tenacity transient-error policy, see above), `MAX_ITERATIONS`, `STRUCTURED_MAX_RETRIES` (default `1`; extra clarification re-prompts on a schema-validation failure before `generate_structured` gives up ŌåÆ `give_up`; `0` disables).
- **`vdb_settings`** (`src/vectordb/config.py`) ŌĆö the vectordb package owns its own knobs. Six are per-project overridable in the web UI (Indexing settings dialog ŌåÆ `meta.json`): the index-time `CHUNK_SIZE`, `CHUNK_OVERLAP`, `DESCRIPTIONS_ENABLED`, `DESCRIBE_MAX_CHARS` (apply = full reindex) and the search-time stitching `EXPAND_PADDING`, `BRIDGE_GAP` (apply = next search, threaded via state `stitch_settings`); the `.env` values below stay the defaults:

| Env var | Default | Meaning |
| --- | --- | --- |
| `LANCE_DB_PATH` | `./data/lancedb/_cli` | CLI/global DB dir (web overrides per project) |
| `EMBEDDING_MODEL` | `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` | FastEmbed model (multilingual, 384d). **Changing the model ŌåÆ full reindex required** |
| `EMBEDDING_CACHE_DIR` | `./data/fastembed_cache` | where FastEmbed stores the downloaded ONNX model (~252MB); the library default `/tmp/fastembed_cache` is wiped on reboot |
| `CHUNK_SIZE` | `1000` | chunk target (chars); new docs only |
| `CHUNK_OVERLAP` | `150` | chunk overlap (chars) |
| `DESCRIPTIONS_ENABLED` | `true` | generate LLM file summaries at index time + Planner routes with them |
| `DESCRIBE_MAX_CHARS` | `6000` | leading chars of each file sent to the LLM for its description |
| `SEARCH_TOP_K` | `5` | nearest chunks per (collection, query) before stitching |
| `OCR_SERVER_URL` | _(unset)_ | delegate LiteParse OCR to a local HTTP sidecar (EasyOCR `:8828` / PaddleOCR `:8829`) instead of built-in Tesseract ŌĆö better Cyrillic, no "image too small" noise, offline. Unset ŌåÆ Tesseract |
| `OCR_LANGUAGE` | _(unset)_ | language code (e.g. `ru`) forwarded to the OCR sidecar |

## Iteration loop

Sufficient Context Agent returns:
- Sufficient ŌåÆ `Command(goto="synthesis")`
- Insufficient + iters left ŌåÆ `Command(goto="planner")` with `feedback`, `missing_parts` (re-route)
- Insufficient + max iters ŌåÆ `Command(goto="give_up")` (system refusal, no LLM)

`feedback` is an **information gap**, not a route ŌĆö recommended form *┬½ąØąĄ čģą▓ą░čéą░ąĄčé: ŌĆ”. ąØą░ą╣ą┤ąĄąĮąŠ ą▓ą╝ąĄčüčéąŠ čŹčéąŠą│ąŠ: ŌĆ”. ąÉą╗čīčéąĄčĆąĮą░čéąĖą▓ąĮčŗąĄ čäąŠčĆą╝čāą╗ąĖčĆąŠą▓ą║ąĖ: ŌĆ”.┬╗* (guidance, not validated), no collection names (this IS validated; see Judge Ōåö Planner separation of concerns). It is required whenever `sufficient=False` ŌĆö the Planner's iteration mode keys off it.

**Re-routing on iteration.** The loop re-enters at the **Planner**: with `feedback` + `iteration_count > 0` set, `planner` uses `PLANNER_ITERATION_PROMPT` to re-route to the collection(s) most likely to hold the missing piece ŌĆö matching the gap (and its alternative phrasings) against collection descriptions and the coverage/novelty statistics, preferring unsearched collections; a collection whose last search yielded `+0 ąĮąŠą▓čŗčģ` may be revisited only with a radically different angle. On the initial turn the Planner may not refuse unsearched (it probes instead ŌĆö see Graph flow); empty steps lead to `give_up` only for an empty knowledge base or, on iteration, when every plausible collection is exhausted (with an exhaustion-specific reason). `query_rewriter` therefore has a single mode: rewrite the Planner's routes, one search task per route, avoiding the already-executed queries it is shown. Max iterations: 3.
