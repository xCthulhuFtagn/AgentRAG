# CLAUDE.md

Agentic RAG — multi-agent retrieval pipeline on LangGraph + DeepSeek + LanceDB.

## Architecture

**Fully edgeless LangGraph graph.** All routing via `Command(goto=...)` returned from nodes. No `add_edge`, no `add_conditional_edges`, no `Send`.

Entry: `set_entry_point("planner")`
Exit: `Command(goto=END)` in synthesis or give_up nodes.

**Pure RAG — single functionality.** Every query goes through retrieval. There is
no orchestrator / complexity gate (no "answer directly without searching") and no
fallbacks: no general-knowledge answer, no answering from noise. Refusals are
honest AND evidence-based: a claim of absence requires an actual search —
descriptions can justify routing priority, never absence — so on the initial
turn the Planner probes the least-implausible collections instead of refusing;
`give_up` without a single search happens only when the knowledge base is empty.

## Graph flow

```
  planner → query_rewriter → search_fanout → sufficient_context
     ▲   │                                       │
     │   └─ empty KB, or iteration exhausted ──► give_up ──► END
     │       insufficient + iters left           │
     └───────────────────────────────────────────┘
     (re-route: planner re-plans for the missing piece)
                                                 │
                       sufficient ──► synthesis ──► END
                       insufficient + max iters ──► give_up ──► END
```

The loop re-enters at **planner** — on iteration the Planner re-routes to the
collection(s) most likely to hold the missing piece (mirrors Google RAG Engine's
loop that re-enters before its Search Plan agent). On the initial turn the
Planner never refuses an non-empty corpus unsearched: if no collection looks
relevant it emits probe routes (prompt rule + a code backstop capped at
`_PROBE_LIMIT`, the candidates ranked by `_rank_collections_for_probe` —
cosine similarity between the query and each collection's description via the
same local FastEmbed model used for chunks, no LLM call — rather than picking
the first `_PROBE_LIMIT` in listing order; collections with no description, or
any embedding failure, fall back to listing order); the judge then rules on
the actual retrieval. It goes straight to `give_up` only with an empty
knowledge base, or on iteration when every plausible collection is searched
to exhaustion.

## Nodes (6 total)

| # | Node | Returns |
|---|------|---------|
| 1 | planner | `Command(goto="query_rewriter" \| "give_up")` — `"give_up"` only for an empty KB or iteration exhaustion; an implausible-looking corpus is probed on the initial turn, not refused |
| 2 | query_rewriter | `Command(goto="search_fanout")` — always rewrites the Planner's routes (one task per route) |
| 3 | search_fanout | `Command(goto="sufficient_context")` |
| 4 | sufficient_context | `Command(goto="synthesis" \| "planner" \| "give_up")` — picks one `verdict` (closed Literal): synthesis verdicts → answer, continue verdicts → `"planner"` re-route, `ничего_не_найдено` → give_up |
| 5 | synthesis | `Command(goto=END)` |
| 6 | give_up | `Command(goto=END)` — system refusal, no LLM |

## Key design decisions

- **No edges** — every node returns `Command(goto=...)`, zero `add_edge` calls
- **No Send** — fan-out parallelism is `asyncio.gather` inside `search_fanout`
- **TypedDict state** — not Pydantic BaseModel; uses `Annotated[list, operator.add]` reducers so `search_results`/`trace`/`rewritten_queries` accumulate across iterations
- **No checkpointer** — `build_graph()` compiles with no checkpointer at all (`workflow.compile()`, no `MemorySaver`). Each query is one independent run with no conversation history to persist across messages, so a checkpointer would only accumulate every past run's full state (every retrieved chunk included) in process memory forever — a real leak in the long-lived web process. Callers pass no `thread_id`/`config` (`graph.astream(state, stream_mode=...)` — CLI and web alike)
- **All async** — nodes are `async def`, tools are `async def`, streaming via `graph.astream()`
- **Only vector search tools** — `vector_search` and `list_collections` via LanceDB; no web/Wikipedia APIs
- **Honest refusal** — Give Up node builds a system-generated message (no LLM) listing what was found, what's missing, and why. The final answer is rendered as **markdown** in the web UI, so collection names (underscores → italics) are backtick-wrapped everywhere they reach the user: give_up wraps its lists and any bare names inside the judge-written reason (`_backtick_names`); Synthesis sees names pre-backticked in its prompt (inventory + source headers) plus an explicit rule, and mirrors the format
- **Structured output via function calling** — `get_structured_llm()` in `common.py` uses `with_structured_output(schema, method="function_calling")`. DeepSeek's API rejects the default json_schema `response_format` ("This response_format type is unavailable now").
- **Validation-driven retries + honest LLM-failure refusal** — every structured node calls `generate_structured(schema, prompt)` (`common.py`), not the raw LLM. Every requirement on a result is a Pydantic constraint (required fields; `RouteStep._non_empty` rejects blank `collection`/`subquery`; `SufficientContextResult._verdict_is_actionable` rejects a `verdict ∈ CONTINUE_VERDICTS` whose `feedback` is empty — its presence is a code contract, the Planner's iteration mode keys off `bool(feedback)` and only continue verdicts route there — or any verdict whose `missing_parts` have blank items; feedback's *form* is deliberately NOT validated: its only consumer is the Planner-LLM reading prose, and a label check would turn good-content-worded-differently into re-prompts that can escalate a correct verdict into give_up). **Every judge field is required — no defaults, flat types.** A weak FC model omits non-required keys outright (observed on GigaChat: a `False` verdict with no `feedback` key), and requiredness is the one schema property weak models obey reliably; conditional emptiness is expressed as a required-but-empty value (`""` / `[]`), never as an optional key. `Optional[...]` would silently undo this on the GigaChat branch: langchain-gigachat's converter **drops nullable fields from the function schema's `required` list** (pinned by `test_gigachat_schema_requires_every_field`, importorskip-guarded here). Constraints needing node-time context use the same path: `make_sufficient_context_schema(collection_names)` (`state.py`) returns a dynamic subclass (function-calling name kept stable) with one extra validator — `feedback`/`missing_parts` of a non-synthesis verdict must not contain any collection name (weak models prescribe routes regularly, so the ban is mechanical, not prompt-only). A violation — or a transport error, or no tool call — raises, and `generate_structured` re-prompts with the failure text up to `STRUCTURED_MAX_RETRIES` times (one uniform path: no per-schema retry hooks). If the model still can't satisfy the schema it raises `StructuredGenerationError`; the `llm_failsafe` wrapper (applied to every node except `give_up` in `build_graph`) catches that **and** `openai.APIError` from the free-text nodes (`query_rewriter`/`synthesis`), routing to `give_up` with `llm_error` set so the refusal honestly cites the model failure instead of crashing. Non-LLM exceptions (code bugs) propagate.
- **All LLM-facing text is Russian** — agent prompts, schema field `description`s (they ship inside the function-calling schema), validator error messages (they are re-prompted back to the model on failure), and the placeholder strings interpolated into prompts (`(без описания)`, `(результатов поиска пока нет)`, …). The corpus is Russian — prompting in English made the model reason in English about Russian documents. Code identifiers, comments, logs and docs stay English.
- **Schema-Guided Reasoning (field order = reasoning order)** — structured output is generated field-by-field in declaration order, so a schema's fields must read like a person's reasoning chain with the verdict near the end, not the start. `SufficientContextResult` (`state.py`) is ordered `reason → draft_answer → missing_parts → retrieval_state → verdict → feedback`: first the analysis (`reason`) of what the chunks say about the question **as asked** — without inflating an underspecified query into sub-questions; then the draft, the gaps; then `retrieval_state` — an SGR **cascade** step that makes the model read the route-availability signals (any unsearched plausible collection? any collection still returning NEW on-topic chunks vs. `+0`?) **before** committing; and only **then** the `verdict` — grounded in the reasoning it just generated. When the verdict was the *first* field, the judge committed up front and rationalized it — letting "not found" drafts pass as sufficient. `feedback` comes last: the information gap is described only after a continue verdict is concluded. The field `description`s also carry the semantics. (A leading `question_verbatim` field — a literal copy of the query, anti-inflation anchor — was removed: the verbatim-copy validator proved brittle (observed on GigaChat: it rewords longer Russian sentences when copying), rejecting an otherwise-correct verdict into a spurious give_up; the anti-inflation job now rides on `reason`'s wording plus the verdict's situation framing, since an underspecified query maps cleanly to `исчерпано_есть_упоминания`.)
- **Verdict is a closed set of retrieval situations, not a bool (SGR Routing)** — the judge emits `verdict: Literal[…]` (`JudgeVerdict` in `state.py`), one of five named situations, never a `sufficient` boolean. A free boolean a weak model has to justify gets rationalized: the model kept reading *"is this thin answer good enough? (no)"* as insufficient even when the corpus was exhausted (the «пушкин» trace burned all iterations inventing "missing biography/works/influence" the user never asked for, ignoring the `+0 новых` exhaustion signal). Classifying into named situations is what weak models do reliably, and each value's wording carries its own criterion, so the call becomes *"which retrieval state am I in?"*. The bool also muddled two orthogonal axes; the situations split them — **route availability** (`есть_необысканная_коллекция`, `есть_неиспробованный_угол`) vs. **what was found** (`ответ_найден`, `исчерпано_есть_упоминания`, `ничего_не_найдено`). Code maps each to one outcome: `SYNTHESIS_VERDICTS` → synthesis (the answer, or the honest *«в источниках об этом есть только …»* over an exhausted-but-thin corpus — off-topic new chunks count as exhaustion too); `CONTINUE_VERDICTS` → planner re-route while a route remains; `ничего_не_найдено` → give_up directly (nothing found AND nowhere left — don't burn the remaining iterations re-searching exhaustion). `есть_неиспробованный_угол` is forbidden for a collection whose last search was `+0` (the signal is baked into the verdict's `description`, so the model can't pick it over diminishing returns). A convenience `sufficient` **property** (`verdict in SYNTHESIS_VERDICTS`) preserves the old binary read for node/trace code. Underspecified queries («пушкин») mean "what does the corpus say about X" → `исчерпано_есть_упоминания` once searched, not a questionnaire. The judge prompt deliberately does NOT receive the iteration counter (budget is code's job; showing it invites "last iteration, so accept" gaming) nor the previous `missing_parts` (echoing them re-anchored the judge on its own earlier inflation).
- **Judge ↔ Planner separation of concerns** — the judge speaks the language of *information*: what was asked, what was found, which fact is missing. `feedback` (when insufficient) is guided toward the template *«Не хватает: …. Найдено вместо этого: …. Альтернативные формулировки: ….»* (prompt/description-level recommendation — only its presence is validated, see above) — and `missing_parts` are short noun phrases naming absent facts. Collection names and "search in X" prescriptions are **validation errors** (see the bound schema above): an early version had the judge name the next collection, which made the Planner a mere executor — it obediently re-searched a known-empty collection twice and couldn't apply its own "prefer unsearched" rule. The judge may still *consider* unsearched plausibly-relevant collections (a negative verdict isn't final while one exists — that's a verdict criterion, so the judge keeps the inventory), but *which* collection to search next is decided by the Planner alone. Alternative phrasings are the judge's to give (only it read the chunks and saw which wording failed); the Planner carries them into `subquery`, the rewriter into the final search query.
- **Mechanical search statistics (code computes, the model reads)** — a weak model can't reconstruct "what has been searched" from collection tags scattered across the retrieved chunks (it hallucinated "not searched yet" for a collection searched on iteration 1). `collection_search_stats` (`agents/common.py`) walks the accumulated `search_results` and the nodes inject pre-formatted ground truth: the **judge** gets the searched set + last-search novelty delta (*«обыскана 3 раза, последний поиск дал +0 новых чанков»* — `+0` is the diminishing-returns exhaustion detector that kills the "let's search there again" loop) **plus the executed queries per collection** (grounds its "is there an untried angle?" call and keeps its «альтернативные формулировки» from repeating tried ones); the **Planner** gets per-collection coverage (*«извлечено 28/210 чанков (13%)»*, totals via `count_chunks` in `vectordb/tools.py`) **plus the same novelty delta** as its stop signal, framed as routing input, never a verdict criterion — its prompt spells out that LOW coverage does NOT mean "barely explored, dig the same spot again" (a similar query re-returns the same top chunks; the Пушкин trace burned 3 iterations on exactly that misreading), and that with every plausible collection exhausted it must return EMPTY steps (→ honest give_up with an exhaustion-specific reason) instead of re-visiting. The judge deliberately gets no percentages — a weak model reads "13% seen" as "87% unexplored → keep searching". The **rewriter** gets the already-executed queries for its collection (`_queries_already_tried`) so iteration rewrites stop converging to the same bag of words. For this record to be complete, `search_fanout` keeps **every** executed search in `search_results`, empty ones included (an empty search is the strongest exhaustion signal); renderers (judge/synthesis/give_up) skip chunkless entries.
- **LLM per-chunk reranking (opt-in) — assessed on hits, before stitching; a failed assessment is unknown, never "irrelevant"** — when a project's `reranking_enabled` is on, `search_fanout` calls the LLM (`assess_chunks_relevance` / `ChunkRelevanceResult`, `agents/common.py`) to judge each retrieved chunk's relevance to the query, feeding `collection_search_stats`'s per-search topic-hit trend (*«прирост по теме: 12/15 → 3/13 → 1/3»*, `format_search_stats_for_judge`). Two properties matter: (1) assessment happens on the raw KNN **hits only**, before `gather_neighbors` stitches in seq-neighbors — a coherent structural block pulled in afterward is never itself judged or amputated chunk-by-chunk; a stitched-in chunk that wasn't a hit gets `relevant=None` (not assessed), never inherits a nearby hit's flag. (2) a chunk's assessment is `bool | None`, not `bool` — `None` means the assessment call itself failed (transport error, no tool call), and is excluded from both the topic-hit numerator and denominator, and — critically — survives `reranking_remove_irrelevant` (only a confirmed `False` is dropped). Treating a failed assessment as `False` would let a transient API hiccup silently discard good context.
- **Logging** — `logged_node` decorator (in `common.py`, applied centrally in `build_graph`) is the single point that emits each node's `trace` entries as `logging` records under the `agentrag` logger. `setup_logging()` (`src/logging_setup.py`) is called by both `src/main.py` and `web/app.py`, so node decisions appear identically under CLI and web. `main.py` prints only the final answer (program output, not a log); the web UI's live trace stream is a separate channel.
- **Per-step token metering** — `logged_node` also meters token usage: it installs a fresh sink (a `_token_sink` `ContextVar`) before each node runs, and `_TokenUsageHandler` (an `AsyncCallbackHandler` attached to every LLM via `get_llm`'s `callbacks=`) adds each call's `prompt_tokens`/`completion_tokens` into the current sink. Works for **structured** calls too (function-calling returns a parsed object with no `usage_metadata`, so a callback is the only place to catch them), and sums across **concurrent** calls (a node's `asyncio.gather` subtasks copy the context, so they share the sink set before the gather). Totals are stamped onto each trace entry as `input_tokens`/`output_tokens` → logged inline (`[in=… out=…]`) and shown in the web UI as a second line under each step plus a per-message Σ total.

## State accumulation & DB scope

- `search_results`, `rewritten_queries`, `trace` use `operator.add` reducer — each Command.update appends. Critical for the iteration loop.
- `search_results` records **every executed search, empty results included** — it is the source the mechanical statistics are computed from (searched set, novelty delta, coverage). Entries that errored carry `"error"` and are excluded from the stats; consumers that render context (judge/synthesis/give_up) skip chunkless entries themselves.
- `search_tasks` (no reducer, overwrite) carries `[{collection, query}]` for the current turn — one concrete-collection task per Planner route. No `collection=None` / search-all mode (pure RAG, no broad fallback).
- `db_path` (set by `make_initial_state(db_path=...)`, default None=global `LANCE_DB_PATH`) scopes every search to one LanceDB → per-project isolation. Threaded into `vector_search`/`list_collections`.
- `stitch_settings` (set by `make_initial_state(stitch_settings=...)`, default None) — per-project **search-time** overrides `{search_top_k, expand_padding, bridge_gap, reranking_enabled, reranking_remove_irrelevant, hybrid_search_enabled}` read by `search_fanout`; None / missing keys → global `vdb_settings`. The web threads the project's saved values here (`web/chat.py`); the CLI passes nothing. (Name kept for history — it carries more than stitching now.)

## Multi-file search

`planner` builds one route per relevant collection; `query_rewriter` emits one `search_task` per route; `search_fanout` searches every `(collection, query)` pair in parallel (`asyncio.gather`). This is what makes cross-file multi-hop work.

## Vector DB (LanceDB)

**LanceDB** — embedded/serverless (no DB process), stores Lance columnar files on disk, async, persists across restarts. Self-contained module at `src/vectordb/`:
- `embeddings.py` — FastEmbed (ONNX, `paraphrase-multilingual-MiniLM-L12-v2`, **384d**, multilingual incl. Russian — an English-only model blinds retrieval on a non-English corpus); `embed`/`embed_batch` run sync ONNX off the loop via `asyncio.to_thread`; model cached `@lru_cache`. The ~252MB model file lives in `EMBEDDING_CACHE_DIR` (default `./data/fastembed_cache`) — FastEmbed's own default is `{tempdir}/fastembed_cache`, and /tmp is tmpfs on many distros: wiped each reboot → re-download every boot, broken air-gapped runs.
- `client.py` — `get_async_db(db_path)` / `get_sync_db(db_path)`; `db_path or LANCE_DB_PATH`. Async connections are cached by `(event loop id, path)` — a single search/judge/planner turn opens several (`vector_search`, `gather_neighbors`, `count_chunks`, `list_collections`), so reusing the connection avoids re-establishing one per call; `invalidate_db_cache(db_path)` drops the cached entry (called before a directory is wiped: `reindex_project`, `ProjectStore.delete`). Sync connections stay uncached (cheap, one-shot indexing runs).
- `config.py` — `VectorDBSettings` (pydantic-settings, `vdb_settings` instance): all vectordb knobs from `.env` (path, model, chunking, search, stitching). See [Configuration](#configuration).
- `describe.py` — `describe_document(text)`: LLM reads an excerpt at index time → a 1–2 sentence content summary. Self-contained (builds its own DeepSeek client from `general_settings`, no `agents` import).
- `descriptions.py` — JSON sidecar (`{db_path}/_descriptions.json`, `table → {file, description}`) storage; `load_descriptions`/`save_descriptions`. Written at index time, read by the Planner.
- `tools.py` — `vector_search(query, collection, top_k, db_path, hybrid=False)` and `list_collections(db_path)` as LangChain `@tool`s. **Async LanceDB gotcha**: `search()` is a coroutine — `q = await table.search(vec)` then `await q.limit(k).to_list()`. Returns chunk `text` + `_distance` (L2, default metric) + `seq` (chunk position). `hybrid=True` additionally queries the table's full-text index (`table.query().nearest_to_text(query)`) and fuses both ranked lists via `_rrf_merge` (Reciprocal Rank Fusion — see Hybrid search below); an FTS query error (no index) silently falls back to vector-only. `gather_neighbors(collection, hit_seqs, …)` does the neighbor stitching (filter-scan by `seq`, no vector). `count_chunks(collection, db_path)` → total rows (or None), feeding the Planner's coverage statistic.
- `indexer.py` — `index_documents(dir, db_path, settings=…)` (from-scratch run over a dir) delegates to `index_files(files, db_path, fresh=…, settings=…)`; `fresh=False` is the **incremental** mode (web file edits): existing tables stay, each given file's previous table — located via the sidecar's file→table mapping, `safe_table_name(stem)` fallback for legacy sidecar-less tables — is dropped up front and rebuilt, and the sidecar is merged, not overwritten; `remove_files_from_index(names, db_path)` drops deleted files' tables + sidecar entries. `settings` = per-corpus overrides resolved by `resolve_index_settings` — absent/None keys fall back to `vdb_settings` (.env stays the default). The resolver covers the four index-time ones (`INDEX_TIME_KEYS` — `chunk_size`, `chunk_overlap`, `descriptions_enabled`, `describe_max_chars`; changing any invalidates stored chunks → full reindex) plus the search-time ones (`search_top_k`, `expand_padding`, `bridge_gap`, `reranking_enabled`, `reranking_remove_irrelevant`, `hybrid_search_enabled`) that ride along in the same per-project dict but are consumed by `search_fanout`, not the indexer. Hybrid extraction (LiteParse for PDF/DOCX/PPTX, `read_text` for TXT/MD) → `clean_text` (collapse ragged whitespace) → `split_text` (RecursiveCharacterTextSplitter, chunk_size/overlap chars, splits on para→line→sentence→word, never mid-word) → `embed_batch` → rows `{text, vector, seq}` → `table.create_fts_index("text", …, language="Russian", stem=True)` (native LanceDB full-text index, built unconditionally — cheap, no LLM — so a project's `hybrid_search_enabled` can be flipped on later without a reindex; a build failure is logged and does not break indexing, same as a legacy table with no index). Per-file work (extraction/embedding/description/table write) runs concurrently across files, bounded by `INDEX_CONCURRENCY` (`asyncio.Semaphore`) — only the table-name collision check stays sequential (it depends on a `used`-names set built incrementally over the run). CLI: `python -m src.vectordb.indexer --dir docs/sample_docs`. LiteParse OCRs scanned/image pages with built-in Tesseract by default; set `OCR_SERVER_URL` (+ `OCR_LANGUAGE`) to delegate to a local EasyOCR/PaddleOCR sidecar via `_get_parser()` — better Cyrillic, silences Tesseract's "Image too small to scale!!" native stderr noise.

**Schema & layout:** one **file → one table** (collection); rows are `{text: str, vector: float[384], seq: int}` (`seq` = chunk index in the document, enables neighbor stitching). Table name = sanitized file stem via `safe_table_name()` (LanceDB allows only `[A-Za-z0-9._-]`; Cyrillic transliterated, hash fallback, collisions disambiguated — in incremental mode against the *existing* table set, not just the current run's). A (re)indexed file's table is always `drop_table` + `create_table` (no row-level upsert), but indexing itself is per-file incremental: only the files passed to `index_files` are touched. No ANN index built → exhaustive search (fine at doc scale).

**Neighbor stitching (deterministic context expansion):** vector search returns the top-k *most similar* chunks, but a contiguous structural block (table of contents, reference list) splits across chunks where only the head ranks high — the tail falls below top-k and the answer truncates. After KNN, `search_fanout` calls `gather_neighbors`: each hit's `seq` becomes a window `[seq-EXPAND_PADDING, seq+EXPAND_PADDING]`; windows merge when the uncovered gap between them is `≤ BRIDGE_GAP` (effective merge distance `2*P + gap + 1`); every chunk in the merged ranges is fetched by `seq` filter-scan and the result is seq-ordered, capped at `MAX_EXPANDED`. Padding and gap are per-project overridable (state `stitch_settings` → `gather_neighbors` kwargs); `MAX_EXPANDED` stays global. Legacy tables without `seq` no-op (reindex to activate). Both the judge and Synthesis render `search_results` via `render_search_context` (`agents/common.py`): grouped by collection, each chunk seq-tagged (so the judge sees contiguity and gaps) and deduplicated by `seq` (legacy tables dedupe by exact text) — a chunk re-fetched by a later iteration's search, or re-pulled by stitching, is shown once, not once per search that returned it.

**Hybrid search (BM25 + vector, opt-in, default on):** vector similarity misses exact terms, names and abbreviations — precisely what a rephrased query is trying to recover, and what drives the judge's "try an alternative phrasing" iteration loop. When a project's `hybrid_search_enabled` is on, `vector_search` (`vectordb/tools.py`) additionally queries the table's native LanceDB full-text index (`table.query().nearest_to_text(query)`, built at index time — see `indexer.py`, Russian stemmer, `use_tantivy=False`) and fuses the two ranked lists via Reciprocal Rank Fusion (`_rrf_merge`, k=60) rather than comparing their raw scores (L2 distance vs. a BM25-like score live on incomparable scales). Dedup key is `seq` (both lists come from the same table) or the chunk's text for legacy tables. A table with no FTS index (legacy, or a failed index build) makes the FTS query raise — caught and silently treated as vector-only, same as `hybrid_search_enabled=False`. The toggle is search-time (`stitch_settings`), but the underlying FTS index is index-time: flipping it on for an already-indexed project needs a reindex to build the index (it is otherwise built unconditionally on every index run, so turning hybrid on later never needs one going forward).

**Per-file descriptions:** at index time `describe_document` (LLM) summarizes each file in 1–2 sentences; stored in `{db_path}/_descriptions.json`. The Planner reads them via `list_collections_described(db_path)` → `[{collection, description}]`, so routing sees a content summary, not just the table name. Legacy tables (indexed before the feature) → empty description; reindex to populate.

**Corpus inventory → judge & synthesis:** the same `list_collections_described` list is also injected (as `get_inventory_str` in `agents/common.py`) into the **Sufficient Context** and **Synthesis** prompts as the *complete, authoritative* inventory of the knowledge base. This closes an epistemic gap: vector search returns similar chunks but never proves it has seen every document, so for "describe/list ALL files"-type queries the judge could never confirm completeness and the loop always ended in `give_up`. With the ground-truth inventory the judge can confirm full coverage (every collection searched or summarized) and Synthesis can describe every file from its summary even where chunks are thin.

The judge uses the inventory **two ways**, and the prompt must keep them distinct (an early version conflated them and made the judge mark a "not found" answer as sufficient): (1) *"all files"* questions → coverage is complete once every collection is searched or summarized — don't demand proof of more documents. (2) *Specific* questions ("what is X") → if the answer isn't in the retrieved chunks, compare the inventory against the **code-computed search statistics** (the judge receives the searched set explicitly — reconstructing it from the collection tags on result blocks is exactly what a weak model gets wrong) and, if any unsearched collection could plausibly hold the answer, return `есть_необысканная_коллекция` — *without naming it*: the existence of an unsearched plausibly-relevant collection means the negative verdict isn't final yet, but choosing the next collection is the Planner's job (naming one in `feedback`/`missing_parts` is a validation error). A negative answer is only final once every plausibly-relevant collection has actually been searched. The **Planner's iteration prompt** receives the same statistics enriched with coverage (`извлечено K/N чанков (P%)`) — it never sees `search_results`, so without an explicit searched-set it would re-route to a known-empty collection and burn iterations.

**Storage & isolation:** everything under `data/` (auto-created by `get_async_db`/`get_sync_db` via `mkdir(parents=True)` and by `ProjectStore`). CLI uses global `LANCE_DB_PATH` = `./data/lancedb/_cli` (`_cli` can't collide with project UUIDs and isn't listed as a project). Web gives each project its own dir `data/lancedb/{project_id}/` and threads that `db_path` through state → `vector_search`/`list_collections`, so a project searches only its own files. File edits update the index **incrementally** (only the delta is re-embedded); full wipe-and-rebuild of `data/lancedb/{id}` happens only when the project's indexing settings change. Data persists between runs.

## web module (NiceGUI)

`web/` is a sibling top-level package — absolute imports `from src... import ...`, one-directional (`web → src`). Run: `python -m web.app`.
- `app.py` — NiceGUI UI passed as `ui.run(root=index)` (script mode needs a root function, not `@ui.page`). Green theme; chat freezes (blue + tremble via inlined `static/style.css`) while a project (re)indexes. `commit_edit` computes the staged-edit **delta** (`added` = new uploads/replacements/rename targets, `removed` = deletions/rename sources) and applies it directly via `ProjectStore`'s per-file operations — `delete_file` for removals, then `rename_file` for renames, then `add_file` for uploads (that order matters: a delete/rename must free a name before anything else claims it) — rather than reading every untouched file into memory and rewriting the whole directory, so a failure only affects the files actually touched. Renames go through a unique temporary name first (`.__rename_tmp_<uuid>`, same suffix so `rename_file`'s extension check still passes): a straight old→new rename silently overwrites `new` on a plain filesystem rename, which breaks when `new` is itself the source of ANOTHER staged rename in the same commit (e.g. `x.txt→y.txt` alongside `y.txt→z.txt`) — the temp hop makes the two-phase application order-independent. No changes → no reindex at all. The project ⋮ menu carries two project-styled dialogs (`.settings-card`), split by cost. **Indexing settings** holds only the four index-time hyperparameters (`INDEX_TIME_KEYS`); applying any change always rebuilds the project index (validates `chunk_overlap < chunk_size`, saves via `set_index_settings`, runs `trigger_reindex`) so its Apply reads *Apply — full reindex*. **RAG process settings** holds the search-time knobs — `search_top_k` (k blocks), the two stitching ones (`expand_padding`, `bridge_gap`), `hybrid_search_enabled` (BM25 fusion — a toggle switched on for an already-indexed project needs a reindex to build its FTS index), the two reranking toggles (`reranking_enabled`, `reranking_remove_irrelevant`) and `max_iterations` (the planner→judge loop budget) — saved only, applied from the next search, never a reindex. Each dialog's Apply button is hidden until the values differ from the saved ones (revert hides it again); plain close discards. Both build their result via `collect()` → `{**current, …}`, so a save touches only that dialog's own keys and never resets the other dialog's overrides back to the `.env` defaults (`set_index_settings` → `resolve_index_settings` fills any absent key from `vdb_settings`).
- `projects.py` — `ProjectStore`: filesystem CRUD. `data/projects/{id}/{meta.json,files/}` + `data/lancedb/{id}/`. File list read from disk (no drift). `get_index_settings`/`set_index_settings`: per-project indexing hyperparameters under `meta.json`'s `index_settings`, always resolved through `resolve_index_settings` so unset projects get the `.env` defaults.
- `runtime.py` — `GRAPH` (built once), `STORE`, per-project status + `asyncio.Lock`. Progress: `start_progress` (full reindex — everything pending) vs `init_partial_progress` (incremental — only the delta spins; untouched files show done, earlier failure flags survive).
- `indexing.py` — `reindex_project(id)`: wipe `data/lancedb/{id}`, re-run `index_documents` (used on settings apply — old chunks were cut with old settings); `update_project_index(id, added, removed)`: incremental — `remove_files_from_index` for removed, `index_files(fresh=False)` for added. Both pass the project's `get_index_settings` and hold the per-project lock; status `reindexing`→`idle`.
- `chat.py` — `run_chat(project_id, query)`: streams `(trace|answer)` from `graph.astream` (the graph carries no checkpointer — see Architecture — so there is no thread/state to manage per message). Threads the project's search-time overrides into `make_initial_state(stitch_settings=…)` so `search_fanout` uses the project's `search_top_k` / `expand_padding` / `bridge_gap` / reranking toggles / `hybrid_search_enabled` instead of the globals. Wraps the stream in a try/except: a non-LLM exception (LLM failures already route to `give_up` inside the graph) is logged and yielded as an honest Russian error answer instead of leaving the chat message stuck on its spinner forever.

## DeepSeek API

OpenAI-compatible endpoint at `https://api.deepseek.com/v1`. Model: `deepseek-chat`. Key from VSCode settings → `.env`. LLM factory cached via `@lru_cache` in `src/agents/common.py`.

**Transient-error retries (tenacity)** — `src/llm_retry.py` is a neutral module (importable by both `agents` and `vectordb` without breaking describe.py's no-`agents` rule) exposing `ainvoke_with_retry(runnable, input)`: a tenacity `AsyncRetrying` policy used at every LLM call site (`generate_structured`, `query_rewriter`, `synthesis`, `describe_document`). Retries **only** transient errors — `openai.RateLimitError` (429: `query_rewriter`'s `asyncio.gather` over routes can burst past the account's rate limit), `openai.InternalServerError` (5xx), `openai.APIConnectionError`/`httpx.TransportError` (connection drops, timeouts) — with exponential backoff + jitter (initial `DEEPSEEK_RETRY_BACKOFF_FACTOR`, cap 60s), honoring a 429's `Retry-After`; `DEEPSEEK_CONNECTION_RETRIES` attempts (default 3, `0` disables). Auth/4xx/validation fail fast. Exhausted retries reraise → `llm_failsafe` → honest `give_up`. Distinct from `STRUCTURED_MAX_RETRIES`, which re-prompts on *semantic* (schema-validation) failures — tenacity absorbs transport noise inside each semantic attempt.

## Configuration

Settings are **pydantic-settings** `BaseSettings` classes — typed, validated, read from `.env` / process env (env var = UPPERCASE field name, case-insensitive). Two objects, two scopes:

- **`general_settings`** (`src/config.py`) — DeepSeek + agent loop: `DEEPSEEK_API_KEY`, `DEEPSEEK_BASE_URL`, `DEEPSEEK_MODEL`, `DEEPSEEK_CONNECTION_RETRIES`/`DEEPSEEK_RETRY_BACKOFF_FACTOR` (tenacity transient-error policy, see above), `MAX_ITERATIONS`, `STRUCTURED_MAX_RETRIES` (default `1`; extra clarification re-prompts on a schema-validation failure before `generate_structured` gives up → `give_up`; `0` disables).
- **`vdb_settings`** (`src/vectordb/config.py`) — the vectordb package owns its own knobs. Several are per-project overridable in the web UI (→ `meta.json`), split across two dialogs: the index-time `CHUNK_SIZE`, `CHUNK_OVERLAP`, `DESCRIPTIONS_ENABLED`, `DESCRIBE_MAX_CHARS` (Indexing settings dialog, apply = full reindex) and the search-time `SEARCH_TOP_K`, `EXPAND_PADDING`, `BRIDGE_GAP`, `HYBRID_SEARCH_ENABLED` plus the reranking toggles and `MAX_ITERATIONS` (RAG process settings dialog, apply = next search, threaded via state `stitch_settings`); `INDEX_CONCURRENCY` is global only (not per-project). The `.env` values below stay the defaults:

| Env var | Default | Meaning |
| --- | --- | --- |
| `LANCE_DB_PATH` | `./data/lancedb/_cli` | CLI/global DB dir (web overrides per project) |
| `EMBEDDING_MODEL` | `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` | FastEmbed model (multilingual, 384d). **Changing the model → full reindex required** |
| `EMBEDDING_CACHE_DIR` | `./data/fastembed_cache` | where FastEmbed stores the downloaded ONNX model (~252MB); the library default `/tmp/fastembed_cache` is wiped on reboot |
| `CHUNK_SIZE` | `1000` | chunk target (chars); new docs only |
| `CHUNK_OVERLAP` | `150` | chunk overlap (chars) |
| `DESCRIPTIONS_ENABLED` | `true` | generate LLM file summaries at index time + Planner routes with them |
| `DESCRIBE_MAX_CHARS` | `6000` | leading chars of each file sent to the LLM for its description |
| `SEARCH_TOP_K` | `5` | nearest chunks per (collection, query) before stitching |
| `HYBRID_SEARCH_ENABLED` | `true` | fuse vector KNN with full-text (BM25-style) search via RRF — recovers exact terms/names/abbreviations. Needs the FTS index built at index time; toggling on for an already-indexed project needs a reindex |
| `INDEX_CONCURRENCY` | `3` | files processed concurrently by the indexer (`asyncio.Semaphore`) — global only, not per-project |
| `OCR_SERVER_URL` | _(unset)_ | delegate LiteParse OCR to a local HTTP sidecar (EasyOCR `:8828` / PaddleOCR `:8829`) instead of built-in Tesseract — better Cyrillic, no "image too small" noise, offline. Unset → Tesseract |
| `OCR_LANGUAGE` | _(unset)_ | language code (e.g. `ru`) forwarded to the OCR sidecar |
| `EXPAND_PADDING` | `1` | neighbor stitching: window `[seq-P, seq+P]` per hit |
| `BRIDGE_GAP` | `2` | merge windows when uncovered gap ≤ this |
| `MAX_EXPANDED` | `16` | cap on stitched chunks per result |

All have defaults — only `DEEPSEEK_API_KEY` is required. Access values via the objects (`vdb_settings.search_top_k`), never module-level constants. Validation rejects bad values (e.g. `SEARCH_TOP_K=0` → `ge=1` error) at startup.

## Iteration loop

Sufficient Context Agent returns a `verdict` (closed Literal), which code maps to an outcome:
- `ответ_найден` / `исчерпано_есть_упоминания` (`SYNTHESIS_VERDICTS`) → `Command(goto="synthesis")`
- `есть_необысканная_коллекция` / `есть_неиспробованный_угол` (`CONTINUE_VERDICTS`) + iters left → `Command(goto="planner")` with `feedback`, `missing_parts` (re-route)
- `ничего_не_найдено`, or a continue verdict at max iters → `Command(goto="give_up")` (system refusal, no LLM)

`feedback` is an **information gap**, not a route — recommended form *«Не хватает: …. Найдено вместо этого: …. Альтернативные формулировки: ….»* (guidance, not validated), no collection names (this IS validated; see Judge ↔ Planner separation of concerns). It is required whenever `verdict ∈ CONTINUE_VERDICTS` — the Planner's iteration mode keys off it.

**Re-routing on iteration.** The loop re-enters at the **Planner**: with `feedback` + `iteration_count > 0` set, `planner` uses `PLANNER_ITERATION_PROMPT` to re-route to the collection(s) most likely to hold the missing piece — matching the gap (and its alternative phrasings) against collection descriptions and the coverage/novelty statistics, preferring unsearched collections; a collection whose last search yielded `+0 новых` may be revisited only with a radically different angle. On the initial turn the Planner may not refuse unsearched (it probes instead — see Graph flow); empty steps lead to `give_up` only for an empty knowledge base or, on iteration, when every plausible collection is exhausted (with an exhaustion-specific reason). `query_rewriter` therefore has a single mode: rewrite the Planner's routes, one search task per route, avoiding the already-executed queries it is shown. Max iterations: 3.
